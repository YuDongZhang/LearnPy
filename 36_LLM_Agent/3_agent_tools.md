# 3. 工具系统详解

## 什么是工具系统

工具（Tool）是Agent与外部世界交互的桥梁，让LLM不再局限于文本生成，可以：
- 搜索实时信息
- 调用API
- 执行代码
- 读写文件
- 查询数据库

## 工具的组成

每个工具需要三个要素：
1. **name** — 工具名称，供LLM识别
2. **description** — 功能描述，LLM据此决定何时使用
3. **func** — 实际执行的函数

> description写得好不好，直接决定LLM能不能正确选择工具。

## 工具类型一览

| 类型 | 功能 | 示例 |
|------|------|------|
| 搜索 | 获取实时信息 | DuckDuckGo、Google |
| API | 调用外部服务 | 天气、股票、邮件 |
| 计算 | 数学运算 | 计算器、Python REPL |
| 文件 | 文件操作 | 读写本地文件 |
| 数据库 | 数据操作 | SQL查询 |
| 代码 | 执行代码 | Python REPL |

## 定义工具的三种方式

### 方式一：简单函数 + Tool包装
最快捷的方式，适合简单工具。

### 方式二：继承BaseTool
适合需要参数校验、复杂逻辑的工具。通过`args_schema`定义输入参数。

### 方式三：异步工具
实现`_arun`方法，适合IO密集型操作（网络请求等）。

## 常用预置工具

### 搜索工具
- `DuckDuckGoSearchRun` — 免费，无需API Key
- `SerpAPIWrapper` — Google搜索，需要API Key
- `WikipediaQueryRun` — Wikipedia搜索

### 代码执行
- `PythonREPLTool` — 执行Python代码并返回结果

### 文件操作
- `FileManagementTool` — 读写本地文件

## 工具描述的最佳实践

好的描述：
```
"获取指定城市的当前天气信息。输入: 城市名称(字符串)。输出: 天气状况、温度、湿度。"
```

差的描述：
```
"查天气"
```

原则：
- 说清楚工具做什么
- 说清楚输入是什么格式
- 说清楚输出是什么

## 工具错误处理

工具内部应该做好异常捕获，返回有意义的错误信息，让LLM可以据此调整策略：
- `ValueError` → 提示输入格式错误
- `TimeoutError` → 提示超时重试
- 通用异常 → 返回错误描述

## 工具安全

- API Key等敏感信息用环境变量，不要硬编码
- 对工具的输入做校验和过滤
- 必要时加权限控制，防止越权操作
