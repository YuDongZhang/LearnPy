# 2. Agent工作流编排

## 什么是Agent工作流

Agent工作流是将Agent的执行过程结构化为可控的流程。与普通Agent"自由决策"不同，工作流预定义了执行步骤和流转规则，在保持灵活性的同时提供更强的可控性。

| 对比 | 普通Agent | Agent工作流 |
|------|----------|------------|
| 执行方式 | 完全动态决策 | 预定义流程+动态决策 |
| 可控性 | 低 | 高 |
| 可预测性 | 低 | 高 |
| 适用场景 | 开放式任务 | 有明确步骤的任务 |

## 四种工作流模式

### 顺序工作流
```
Step1 → Step2 → Step3 → 结果
```
最简单的模式，步骤按顺序执行。

### 并行工作流
```
      ┌→ Step1 ─┐
输入 ──┼→ Step2 ──┼→ 合并 → 输出
      └→ Step3 ─┘
```
独立步骤并行执行，提高效率。

### 条件工作流
```
输入 → 判断 ─→ 条件A → StepA
             └→ 条件B → StepB
```
根据条件选择不同的执行路径。

### 循环工作流
```
生成 → 检查 ─→ 满足 → 输出
        └→ 不满足 → 反馈 → 生成（重来）
```
反复迭代直到满足质量要求。

## 工作流设计原则

1. **单一职责** — 每个步骤只做一件事
2. **清晰接口** — 步骤之间的数据格式明确
3. **可观测性** — 记录每步的输入输出，便于调试
4. **错误处理** — 每步有异常捕获，支持重试和降级

## 状态管理

工作流需要在步骤之间传递和维护状态：
- **内存状态** — 用字典或TypedDict保存，适合短流程
- **持久化状态** — 存入数据库或文件，适合长流程、可恢复

## 错误处理策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 重试 | 失败后自动重试N次 | 网络请求、API调用 |
| 降级 | 主方案失败用备选方案 | 有替代方案时 |
| 跳过 | 跳过失败步骤继续 | 非关键步骤 |
| 中断 | 停止整个工作流 | 关键步骤失败 |

## 监控与日志

生产环境的工作流需要：
- 每步执行时间记录
- 输入输出日志
- 异常告警
- 性能指标统计

## LCEL（LangChain Expression Language）

LangChain提供的链式工作流语法，用 `|` 管道符连接步骤：
```python
chain = prompt | llm | output_parser
```
适合简单的线性工作流。复杂工作流推荐用LangGraph。
